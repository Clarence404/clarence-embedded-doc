# Linux 与 MCU 的区别

> 承接 [技术全景](/overview/0_layered_arch)：这两层是四层链的中间两环，也是最容易被搞混的一对。

一句话：**MCU 是"一段直接跑在芯片上的程序"，Linux 是"一个能跑很多程序的操作系统"**。两者相差的是一整个数量级的复杂度。

## 对比表

| 维度 | MCU（如 STM32） | Linux（跑在 MPU 上，如 i.MX6ULL） |
|---|---|---|
| 有无操作系统 | 裸机 或 RTOS（FreeRTOS） | 完整 Linux，有进程、虚拟内存 |
| 内存 | KB 级（几十~几百 KB） | MB~GB 级 |
| 有无 MMU | **无**内存管理单元 | **有**，进程隔离、虚拟地址 |
| 实时性 | 硬实时，微秒级确定 | 软实时，有调度抖动 |
| 启动速度 | 毫秒级 | 秒级（要加载内核 + 文件系统） |
| 开发方式 | 单个固件镜像，操作寄存器 | 多进程 + 文件系统 + 驱动/应用分层 |
| 功耗 | 极低（μA 级休眠） | 高得多 |
| 联网能力 | 需外挂模块，协议栈简陋 | 原生 TCP/IP，天然联网 |
| 擅长 | 实时控制、高频采样、低功耗 | 网络、复杂协议、数据处理、上云 |

## 电表场景里的分工

在计量设备里，两者的分工可以这样记：

- **MCU 负责"测得准"**——以极高频率采样电压电流、算功率算电能，硬实时、不容一丝延迟抖动。这是计量核心，通常还涉及专用计量芯片和法定计量认证。这活儿 Linux 干不了，调度抖动会毁掉精度。
- **Linux 负责"传得出、管得好"**——从 MCU 收数据、做协议转换（Modbus → MQTT）、联网上云、远程升级、多设备管理。这活儿 MCU 很吃力，内存不够、协议栈太重。

## 用 Java 经验类比

- **Linux** ≈ 一台跑着你 Spring Boot 应用的服务器：有操作系统、有进程、有 TCP/IP、有文件系统。你现在的知识**能直接迁移**，只是把 Java 换成 C。
- **MCU** ≈ Java 里**没有对应物**。最接近的想象是：没有 JVM、没有操作系统、没有 GC，你的代码就是全部，直接读写内存地址来控制引脚。

::: warning 这正是选型的依据
MCU 对你的经验复用率最低，所以本站把它定位为**参考模块**（懂即可，见 [mcu](/mcu/0_mcu_intro)）；而 Linux 与你的背景高度契合，是**主线**。
:::

## 选型边界：什么时候用哪个

| 需求 | 选 MCU | 选 Linux |
|---|---|---|
| 微秒级硬实时控制 | ✅ | ❌ |
| 超低功耗 / 电池供电 | ✅ | ❌ |
| 成本极度敏感（￥1~5 芯片） | ✅ | ❌ |
| 复杂网络协议 / 上云 | ❌ | ✅ |
| 跑数据库 / 容器 / 多任务 | ❌ | ✅ |
| 需要文件系统与远程运维 | ❌ | ✅ |

现实中很多产品是**两者搭配**：MCU 管实时采集与计量，Linux 管联网与管理，中间用串口对接——这正是[四层体系](/overview/0_layered_arch)里 MCU ↔ Linux 那一环，也是 [mcu/3_mcu_linux_link](/mcu/3_mcu_linux_link) 要讲的联调重点。
