# 技术全景：云 · Linux · MCU · 硬件四层体系

> 读这个站之前先看这一篇。它解释了所有模块**为什么这样连、你站在哪**。

一个完整的物联网 / 工业设备产品，从物理世界到云端，通常由四层构成。它们是一条**相邻对接链**——关键规律是：**每一层只和它相邻的层直接对话，从不越级**。

```
云平台                          ← Java 后端（你已精通）
  ↕   MQTT / HTTP                   北向：联网上报、远程配置、OTA
Linux 网关                      ← 本站主线，你要进的组
  ↕   串口 / Modbus RTU / 自定义协议   中间：协议转换、数据缓存
MCU                             ← 计量、实时采集（南向参考）
  ↕   寄存器 / ADC / GPIO / SPI       芯片级：读取物理量
硬件                            ← 传感器、采样电路、电源（延伸）
  ↕   电压 / 电流 / 模拟信号
物理世界（电流、电压、温度……）
```

## 数据的真实流向

把上图反过来读，就是一次采集上报的完整路径：

```
物理量  →  硬件采样  →  MCU 计量  →  Linux 网关  →  云平台
(电流)    (采样电路)   (算功率/电能)  (协议转换/联网)  (存储/分析/展示)
```

::: tip 核心认知
云端**从不直接**和 MCU 或硬件对话，中间永远隔着 Linux 网关这座桥。这座桥，就是本站要带你占据的位置。
:::

## 每层职责与对接接口

| 层 | 公司对应组 | 职责 | 与上下层的对接接口 |
|---|---|---|---|
| 云平台 | 上位机 / 平台组 | 数据存储、分析、可视化、设备管理 | ↓ 向下：MQTT / HTTP |
| Linux 网关 | **Linux 组** | 协议转换、数据缓存、联网上云、远程升级 | ↑ MQTT ／ ↓ 串口、Modbus |
| MCU | MCU 组 | 高频采样、计量运算、实时控制 | ↑ 串口、Modbus ／ ↓ 寄存器、ADC |
| 硬件 | 硬件组 | 传感器、采样调理电路、电源、PCB | ↑ 寄存器、模拟信号 ／ ↓ 物理量 |

## 不是每个产品都四层齐全

- 简单设备：MCU 直接外挂 NB-IoT / 4G 模块就上云，**跳过 Linux 层**。
- 部分方案：Linux 直接接 ADC 采样，**跳过 MCU 层**。
- 高精度计量设备（如电表）：四层齐全，MCU 层几乎不可省——因为计量需要硬实时，Linux 的调度抖动会破坏精度（原因见 [Linux 与 MCU 的区别](/overview/1_linux_vs_mcu)）。

## 这个文档站怎么对应这条链

全站模块按「和你的转型路线的关系」分三个层级：

| 层级 | 对应链上的层 | 本站模块 | 学习目标 |
|---|---|---|---|
| **主线**（做深做透） | 云 ↔ Linux 网关 | basics / shell / system / c-lang / sys-programming / network / embedded / peripherals / drivers / kernel / protocols / gateway / debugging | 胜任岗位 |
| **延伸**（从懂到能做） | 硬件 | hardware | 能做简单采集板 |
| **参考**（懂即可） | MCU | mcu | 能与 MCU 团队联调 |

云平台那一层的知识体系沉淀在姊妹站 **clarence-java-doc**（iot / messaging / netty 等模块），本站只在 [gateway](/gateway/0_gateway_arch) 做北向衔接。

## 你的位置：Linux 网关

你不需要会画 PCB，也不需要会写 MCU 固件。你要做的是那个**能把「设备到云端」整条链一个人打通**的人：

- **北向**——凭 Java 后端底子，云那半边你本就精通；
- **南向**——懂 MCU 吐出什么协议、懂硬件的信号从哪来，就能和两边高效对话。

这个能力形状全公司可能只有你一个。这也是本站「Linux 为主，硬件与 MCU 为参考」分层的根本原因。
